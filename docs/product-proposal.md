# Product Proposal

## Обоснование идеи

Поиск научных статей является первым шагом практически любого исследования.

Однако этот процесс часто требует значительных временных затрат:

- поисковые системы возвращают большое количество нерелевантных результатов
- пользователь должен вручную анализировать десятки abstract'ов
- сложно определить, какие статьи являются ключевыми для начала изучения темы

AI Research Paper Finder решает эту проблему, автоматизируя первичный этап поиска литературы.

Система использует LLM и внешнию базу знаний для:

- улучшения поисковых запросов
- поиска релевантных статей
- ранжирования результатов
- генерации кратких описаний

Это позволяет значительно сократить время первичного literature search

---

## Цель и метрики

### Цель проекта

Создать агентную систему, которая помогает пользователю быстро находить релевантные научные статьи по заданной теме.

---

#### Продуктовые метрики

- пользователь получает релевантный список статей по теме
- сокращение времени на первичный поиск литературы
- понятность объяснения релевантности статьи

---

### Агентские метрики

- доля релевантных статей в топ-5 результатов - Precision@5
- качество reformulation пользовательского запроса - Precision@5 после query reformulation должна быть выше или равна Precision@5 исходного запроса

---

### Технические метрики

- p95 latency ответа системы < 15 секунд
- успешность вызова внешнего API > 95%
- стабильная работа без критических ошибок

---

## Потенциальные сценарии использования

### Стандартный запрос

Пользователь вводит тему исследования:

"Graph neural networks for traffic prediction"

Система:

1. улучшает поисковый запрос
2. выполняет поиск в arXiv
3. анализирует найденные статьи
4. возвращает список наиболее релевантных работ

---

### Повторный запрос

Пользователь может продолжить поиск, например:

"теперь только статьи после 2022 года"

Система использует сохраненный контекст и обновляет результаты поиска.

---

## Ограничения

### Технические ограничения

- использование внешнего API (например arXiv)
- ограничение на количество запросов
- p95 latency системы ~10–15 секунд
- зависимость от стабильности API

---

### Операционные ограничения

- ограниченный объем обрабатываемых статей
- PoC масштабируется на ограниченное количество пользователей

---

## Архитектурный набросок

Система состоит из нескольких компонентов:

- `Query Agent`  — анализирует пользовательский запрос и улучшает его.
- `Retrieval Tool` — выполняет поиск статей через arXiv API.
- `Ranking Agent` — оценивает релевантность найденных статей.
- `Summarization Agent` — создает краткое описание каждой статьи.
- `Session Memory` — хранит контекст пользователя и предыдущие запросы.
- `Telegram Bot Interface` — обеспечивает взаимодействие пользователя с системой.

---

## Потенциальный Data Flow

1. Пользователь отправляет тему исследования в Telegram-бот
2. Query Agent преобразует пользовательский запрос
3. Retrieval Tool выполняет поиск статей через arXiv API
4. Ranking Agent оценивает релевантность результатов
5. Summarization Agent генерирует краткие описания
6. Система отправляет пользователю список статей

---

### Делегирование задач

LLM используется для:

- reformulation поисковых запросов
- оценки релевантности статей
- генерации summary

Не делегируется LLM:

- выполнение API запросов
- обработка сетевых ошибок
- управление логикой приложения
- управление состоянием системы
