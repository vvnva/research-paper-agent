# System Design

## 1. Назначение системы

AI Research Paper Finder — PoC агентной системы для поиска и первичного анализа научных статей по пользовательской теме исследования.

Система принимает текстовый запрос от пользователя через Telegram-бота, выполняет поиск статей во внешнем источнике, ранжирует найденные результаты, генерирует краткие summary и возвращает пользователю компактную подборку релевантных работ.

Система не предназначена для полного literature review, анализа полных PDF или генерации академических выводов. Основная задача — ускорить начальный этап поиска литературы.


---

## 2. Ключевые архитектурные решения

- Telegram используется как thin client
- Централизованный orchestrator управляет execution flow
- Retrieval-first pipeline (сначала данные, потом LLM)
- Ограниченный session state (без долгосрочной памяти)
- Fail-safe поведение при деградации внешних зависимостей
- Ограничение числа LLM-вызовов и размера контекста

---

## 3. Модули системы

### Telegram Bot Interface
- прием/отправка сообщений
- проксирование запросов в backend

---

### Orchestrator
- управление workflow
- маршрутизация между модулями
- применение retry/fallback правил
- контроль stop conditions

---

### Query Rewriter
- нормализация исходного запроса
- расширение/уточнение query
- формирование search query для retrieval-модуля

---

### Retriever
- вызов arXiv API
- получение кандидатов
- нормализация полей результата
- базовая фильтрация некорректных или пустых записей

---

### Ranker
- оценка соответствия статьи пользовательскому запросу
- reranking кандидатов
- удаление слабых или дублирующихся результатов
- отбор top-N статей для ответа

---

### Summarizer
- формирование краткого summary по title + abstract + metadata
- краткое объяснение релевантности
- формирование ответа

---

### Session State Store
Содержит:
- последний пользовательский запрос
- нормализованный запрос
- активные фильтры
- список последних найденных статей
- timestamp последней активности

---

### Configuration and Secrets
Содержит:
- ключи API
- параметры моделей
- timeout
- retry policy
- лимиты на количество кандидатов

---

### Logging / Observability
- логирование этапов workflow
- фиксация ошибок внешних API
- сбор latency по шагам
- сбор счетчиков успешных и неуспешных запросов

---

## 4. Основной workflow

1. Пользователь отправляет тему исследования в Telegram-бот.
2. Telegram Bot Interface передает запрос в Request Orchestrator.
3. Orchestrator валидирует входной запрос.
4. Query Rewriter формирует улучшенный search query.
5. Retriever вызывает arXiv API и получает список кандидатов.
6. Ranker оценивает релевантность найденных статей.
7. Из результатов выбираются top-N статей.
8. Summarizer генерирует краткие описания.
9. Orchestrator сохраняет session state.
10. Telegram Bot Interface отправляет пользователю ответ.

---

## 5. State / Memory

### Что хранится в state
На уровне сессии сохраняются:
- `user_id` или технический session key;
- `last_user_query`;
- `normalized_query`;
- `active_filters`;
- `last_results`;
- `last_interaction_ts`.

### Политика:
- Память ограничена одной активной пользовательской сессией
- При истечении TTL сессия очищается
- Если контекст отсутствует, follow-up запрос трактуется как новый запрос

---

## 6. Retrieval-контур

- источник: arXiv API
- вход: rewritten query
- выход: список кандидатов (title, abstract, metadata)

Pipeline:
1. Получить пользовательский запрос
2. Выполнить query rewriting
3. Сформировать поисковую строку
4. Вызвать arXiv API
5. Получить список кандидатов
6. Нормализация
7. Отфильтровать пустые и некорректные записи
8. Передать ограниченный набор кандидатов в Ranker


Ограничения:
- Поиск выполняется только по одному внешнему источнику
- Полные PDF не анализируются
- Верхняя граница числа кандидатов фиксируется конфигурацией

## 8. Failure modes и fallback

**Пустой или слишком короткий польовательский запрос** 
- Детект: длина ниже порога  
- Действие: запросить уточнение у пользователя  

**Слишком общий запрос**
- Детект: эвристика / classifier  
- Действие: авто-уточнение или запрос уточнения   

**Ошибка arXiv API**
- Действие:
  1. retry (ограниченный)  
  2. fallback: сообщение пользователю  

**Нерелевантные или повторяющиеся результаты**
- retrieval возвращает шум или дубликаты
- Действие: post-filtering, deduplication, reranking, ограничение числа статей одного типа

**Prompt injection (abstract)**
- Действие:
  - обрабатывать текст как данные  
  - использовать строгий prompt template  

**Галлюцинации**
- Действие:
  - summary только из abstract + metadata  
  - без добавления внешних фактов  

---

## 9. Ограничения

### Latency
- p95 end-to-end latency ≤ 15 секунд

---

### Cost control
- ограничение числа LLM-вызовов на один запрос
- ограничение числа кандидатов на reranking
- ограничение размера контекста

---

### Reliability
- успешность вызова внешнего retrieval API ≥ 95%
- отсутствие критических ошибок на уровне backend ≥ 99%
- корректное деградирующее поведение при отказе LLM

---

## 10. Точки контроля качества

1. **Input validation**  
   Проверка непустого и осмысленного запроса

2. **Post-retrieval validation**  
   Проверка, что найдено достаточное число корректных записей

3. **Post-ranking validation**  
   Проверка, что в финальном списке нет пустых или дублирующихся статей

4. **Post-summary validation**  
   Проверка формата и длины summary

5. **Response validation**  
   Проверка, что пользователю возвращается структурированный и безопасный ответ даже при частичной деградации системы

## 12. Минимальный observability scope

Система должна собирать:

- число входящих запросов
- число успешных ответов
- число частично успешных ответов
- число ошибок по типам
- latency по шагам:
  - query rewrite
  - retrieval
  - ranking
  - summarization
- число fallback-срабатываний
- число пустых результатов
- среднее число найденных кандидатов
