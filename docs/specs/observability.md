# Observability and Evals Specification

## 1. Метрики

Продуктовые
- `precision_at_5`
- доля запросов с хотя бы одной полезной статьей
- среднее время до получения результата

Основные технические:

- `request_count` — число запросов  
- `success_rate` — доля успешных ответов  
- `error_rate` — доля ошибок  
- `fallback_count` — число fallback  
- `p95_latency_total` — время ответа  

По шагам:
- `latency_retrieval`
- `latency_ranking`
- `latency_summarization`

---

## 2. Логи

Логируются:
- начало и конец запроса
- шаги workflow
- типы ошибок
- срабатывания fallback
- число найденных кандидатов
- итоговый статус запроса

Не логируются:
- персональные данные
- полный пользовательский текст
- полный текст abstract
- полный ответ модели

---

## 3. Трейсы

Для каждого запроса trace по шагам:
1. input validation
2. query rewrite
3. retrieval
4. ranking
5. summarization
6. response formatting

Для каждого шага фиксируются:
- start / end time
- статус
- тип ошибки
- fallback flag
---

## 4. Проверки

На этапе тестирования проверяются:
- корректность обработки стандартных запросов
- обработка слишком общих запросов
- обработка слишком узких запросов
- работа follow-up сценариев
- корректность fallback при отказе arXiv API
- корректность fallback при отказе LLM
---

## 5. Eval-запросы

Для PoC используется небольшой eval-set из 20-30 запросов по разным темам:
- NLP
- Computer Vision
- Recommender Systems
- Robotics
- Graph ML

Для каждого запроса оцениваются:
- релевантность top-5
- качество rewrite
- корректность summary
- наличие дубликатов

---

## 6. Алерты

Алерты настраиваются с фиксированными порогами:

- рост доли ошибок внешних API  
  - триггер: `error_rate_external_api > 7%` 

- рост общей доли ошибок  
  - триггер: `error_rate > 5%`

- рост p95 latency  
  - триггер: `p95_latency_total > 25 секунд`  

- рост числа пустых результатов  
  - триггер: `empty_results_rate > 20%` 

- частое срабатывание fallback  
  - триггер: `fallback_rate > 25%`  

При срабатывании алерта:
- фиксируется событие в системе мониторинга  
- отправляется уведомление (alerting channel)  
- требуется ручная проверка состояния внешних зависимостей и LLM  

---

## 7. SLO и Error Budget

### Service Level Objectives (SLO)

Для системы задаются следующие целевые показатели:

- p95 latency ≤ 15 секунд  
- success rate ≥ 95%  
- error rate ≤ 5%  
- доля пустых результатов ≤ 20%  

---

### Действия при нарушении SLO

При превышении error budget:

1. фиксируется инцидент  
2. временно ограничивается нагрузка или функциональность:
   - отключение summarization  
   - уменьшение числа кандидатов  
3. проверяются внешние зависимости:
   - LLM API  
   - arXiv API  
4. анализируются метрики и логи  
5. при необходимости — rollback конфигурации или промптов
