# Serving and Configuration Specification

## Serving

Сервис состоит из следующих runtime-компонентов:
- Telegram bot interface
- backend application
- orchestrator
- session state store
- logging / observability layer

PoC предполагает один backend service без горизонтального масштабирования

---

## Runtime health management
### Health checks

Сервис предоставляет следующие endpoints:

- `/health` — liveness probe  
  - возвращает OK, если процесс запущен  
- `/ready` — readiness probe  
  - проверяет доступность:
    - конфигурации  
    - session state store 

Используется для:
- мониторинга состояния сервиса  
- интеграции с orchestration системами 

---

### Graceful shutdown

При завершении работы сервис должен выполнять controlled shutdown в следующем порядке:

1. перевести сервис в состояние `not ready`  
   - endpoint `/ready` начинает возвращать fail  
   - новый входящий трафик больше не направляется в инстанс  

2. прекратить прием новых запросов  
   - Telegram webhook / HTTP server перестает принимать новые запросы  
   - новые update не передаются в execution pipeline  

3. дождаться завершения in-flight запросов  
   - текущие запросы завершаются в пределах заданного shutdown timeout  
   - если запрос не завершился за допустимое время, он прерывается с безопасным fallback  

4. корректно закрыть внешние соединения  
   - HTTP connections к LLM API  
   - HTTP connections к arXiv API  
   - соединение с session state store 

5. сохранить актуальный session state  
   - сохраняется только уже вычисленное состояние  
   - частично вычисленные промежуточные данные не должны записываться как финальные  

6. завершить процесс  
   - после закрытия соединений и завершения активных задач процесс завершается корректно

## Configuration management
###  Конфигурация

Конфигурация задается через `.env`.

Примеры параметров:
- `TELEGRAM_BOT_TOKEN`
- `LLM_API_KEY`
- `ARXIV_TIMEOUT_SEC`
- `LLM_TIMEOUT_SEC`
- `MAX_CANDIDATES`
- `SESSION_TTL_SEC`
- `RETRY_COUNT`
- `LOG_LEVEL`

---

### Секреты

К секретам относятся:
- Telegram bot token
- LLM API key

Требования:
- не хранить секреты в репозитории
- передавать секреты через environment
- ограничить доступ к production secrets

---

### Версии моделей

Для PoC фиксируются:
- одна основная LLM для rewriting / ranking / summarization
- версия модели задается конфигурацией
- смена модели не должна требовать переписывания бизнес-логики

Дополнительно:
- допускается fallback на альтернативную модель или провайдера при недоступности основного LLM  

---

## Ресурсные ограничения

- ограниченное число LLM-вызовов на запрос
- ограниченный размер контекста
- ограниченное число кандидатов для ranking
- отсутствие batch processing и фоновых workers в первой версии

---

## Надежность запуска

- сервис должен стартовать без session store в degraded mode
- при недоступности LLM сервис не должен падать на уровне процесса
- конфигурация валидируется на старте приложения

---

## Circuit breaker

Для внешних зависимостей используется circuit breaker:

- применяется к:
  - arXiv API  
  - LLM API  

Поведение:
- при превышении порога ошибок (error rate > 50% за интервал в 60 секунд)  
- переход в состояние "open"  
- временное отключение вызовов внешнего сервиса  
- использование fallback-логики  

Цель:
- предотвращение каскадных сбоев  
- снижение нагрузки при деградации внешних сервисов  

---

## Кэширование

Используется базовое кэширование

- кэшируются результаты retrieval (`arXiv API`) по `normalized query`  
- в кэше сохраняются:
  - список найденных статей  
  - metadata (title, abstract, url)  
- TTL кэша ограничен ( 1 час)  
- при повторном запросе с тем же `normalized query` retrieval может не вызываться повторно

## Docker

### Dockerfile (PoC)


Сервис упаковывается в Docker image для обеспечения воспроизводимого runtime-окружения

Dockerfile должен выполнять следующие шаги:

- использовать базовый Python runtime image  
- копировать исходный код приложения  
- устанавливать зависимости  
- настраивать рабочую директорию  
- запускать backend-приложение  

Требования:
- образ должен быть минимальным (только runtime-зависимости)  
- секреты не должны включаться в image  
- конфигурация передается через environment variables 
- должен поддерживаться health endpoint (`/health`)  

---

### Docker image

Image содержит:
- backend application  
- orchestrator  
- зависимости  

Image не содержит:
- `.env` файлы  
- API ключи и другие секреты  
- локальные временные файлы  
---

### docker-compose

Конфигурация включает:

- backend service  
- Redis для session state / cache

Назначение:
- локальный запуск системы  
- упрощение настройки окружения  
- запуск всех зависимостей одной командой  

Требования:
- конфигурация передается через environment variables  
- Redis используется как временное хранилище с TTL  
---

## CI/CD

Pipeline:

1. **Lint** 
   - проверка стиля и базовых ошибок
   - `ruff`, `flake8`
2. **Tests** 
   - запуск unit tests
   - запуск базовых integration tests для ключевых сценариев
3. **Build** 
    - сборка Docker image
    - проверка, что приложение успешно собирается в целевом runtime-окружении
4. **Deploy**  
   - развертывание Docker image в целевое окружение   


## Deployment assumptions

Для PoC допускается:
- локальный запуск
- single-instance deployment
- ручное управление конфигурацией
