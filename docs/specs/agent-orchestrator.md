# Agent Orchestrator Specification



## Основные шаги

1. принять запрос от Telegram interface
2. выполнить input validation
3. загрузить session state
4. вызвать `Query Rewriter`
5. вызвать `Retriever`
6. вызвать `Ranker`
7. вызвать `Summarizer`
8. сохранить обновленный state
9. вернуть ответ пользователю

---

## Правила переходов

### Input validation
- если запрос пустой или слишком короткий — остановить workflow и вернуть просьбу уточнить запрос

### Query rewrite
- если rewrite успешен — использовать rewritten query
- если rewrite недоступен — использовать исходный query

### Retrieval
- если retrieval успешен — передать кандидатов в `Ranker`
- если retrieval вернул пустой список — выполнить один шаг query expansion или вернуть частичный ответ
- если retrieval упал по timeout — выполнить ограниченный retry

### Ranking
- если ranking недоступен — использовать базовый порядок кандидатов
- если результатов недостаточно — вернуть все доступные корректные записи

### Summarization
- если summarization недоступен — вернуть title + url + metadata без summary

---

## Stop conditions

Workflow завершается, если:
- получен валидный ответ пользователю
- входной запрос невалиден
- внешний retrieval API недоступен после всех retry
- число ошибок на шаге превысило допустимый лимит
- превышен глобальный timeout выполнения pipeline

---

## Retry / fallback policy

- retry выполняется только для внешних idempotent вызовов  
- число retry ограничено конфигурацией (по умолчанию: не более 3 попыток)  
- используется экспоненциальный backoff (1с -> 2с -> 4с)  
- retry применяется к:
  - вызовам arXiv API  
  - вызовам LLM API  
- fallback должен отдавать пользователю частично полезный результат, если это возможно
- при срабатывании circuit breaker вызовы внешнего API не выполняются 

---

## Timeout policy

- глобальный timeout pipeline: ≤ 15 секунд 
- таймауты по шагам:
  - query rewrite: ≤ 2 секунды  
  - retrieval: ≤ 5 секунд  
  - ranking: ≤ 3 секунды  
  - summarization: ≤ 5 секунд  
- при превышении timeout на шаге:
  - шаг считается неуспешным  
  - применяется fallback-логика  
- при превышении глобального timeout:
  - workflow прерывается  
  - пользователю возвращается частичный или деградированный ответ 
---

## Выход orchestrator

Финальный ответ должен содержать:
- список статей
- краткие summary или упрощенный fallback-ответ
- понятное сообщение пользователю при частичной деградации
